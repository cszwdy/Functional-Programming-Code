The readFile1 function returns an optional string. It simply calls the initializer, passing the address for an NSError object. If the call to the initializer succeeds, the resulting string is returned; upon failure, the whole function returns nil.

This interface is a bit more precise than Objective-C's stringWithContentsOfFile — from the type alone, it is clear that this function may fail. There is no temptation for developers to pass null pointers for the error argument, ignoring the possibility of failure.

There is one drawback to using Swift's optional type: we no longer return the error message when the file cannot be read. This is rather unfortunate — if a call to readFile1 fails, there is no way to diagnose what went wrong. Does the file not exist? Is it corrupt? Or do you not have the right permissions?
